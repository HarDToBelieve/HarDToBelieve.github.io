51 // Some notes:
52 //
53 // - The canonical way of clobbering the world is to read world and write
54 // heap. This is because World subsumes Heap and Stack, and Stack can be
55 // read by anyone but only written to by explicit stack writing operations.
56 // Of course, claiming to also write World is not wrong; it'll just
57 // pessimise some important optimizations.
58 //
59 // - We cannot hoist, or sink, anything that has effects. This means that the
60 // easiest way of indicating that something cannot be hoisted is to claim
61 // that it side-effects some miscellaneous thing.
62 //
63 // - We cannot hoist forward-exiting nodes without some additional effort. I
64 // believe that what it comes down to is that forward-exiting generally have
65 // their NodeExitsForward cleared upon hoist, except for forward-exiting
66 // nodes that take bogus state as their input. Those are substantially
67 // harder. We disable it for now. In the future we could enable it by having
68 // versions of those nodes that backward-exit instead, but I'm not convinced
69 // of the soundness.
70 //
71 // - Some nodes lie, and claim that they do not read the JSCell_structureID,
72 // JSCell_typeInfoFlags, etc. These are nodes that use the structure in a way
73 // that does not depend on things that change under structure transitions.
74 //
75 // - It's implicitly understood that OSR exits read the world. This is why we
76 // generally don't move or eliminate stores. Every node can exit, so the
77 // read set does not reflect things that would be read if we exited.
78 // Instead, the read set reflects what the node will have to read if it
79 // *doesn't* exit.
80 //
81 // - Broadly, we don't say that we're reading something if that something is
82 // immutable.
83 //
84 // - This must be sound even prior to type inference. We use this as early as
85 // bytecode parsing to determine at which points in the program it's legal to
86 // OSR exit.
87 //
88 // - If you do read(Stack) or read(World), then make sure that readTop() in
89 // PreciseLocalClobberize is correct.
90
91 // While read() and write() are fairly self-explanatory - they track what sorts of things the
92 // node may read or write - the def() functor is more tricky. It tells you the heap locations
93 // (not just abstract heaps) that are defined by a node. A heap location comprises an abstract
94 // heap, some nodes, and a LocationKind. Briefly, a location defined by a node is a location
95 // whose value can be deduced from looking at the node itself. The locations returned must obey
96 // the following properties:
97 //
98 // - If someone wants to CSE a load from the heap, then a HeapLocation object should be
99 // sufficient to find a single matching node.
100 //
101 // - The abstract heap is the only abstract heap that could be clobbered to invalidate any such
102 // CSE attempt. I.e. if clobberize() reports that on every path between some node and a node
103 // that defines a HeapLocation that it wanted, there were no writes to any abstract heap that
104 // overlap the location's heap, then we have a sound match. Effectively, the semantics of
105 // write() and def() are intertwined such that for them to be sound they must agree on what
106 // is CSEable.
107 //
108 // read(), write(), and def() for heap locations is enough to do GCSE on effectful things. To
109 // keep things simple, this code will also def() pure things. def() must be overloaded to also
110 // accept PureValue. This way, a client of clobberize() can implement GCSE entirely using the
111 // information that clobberize() passes to write() and def(). Other clients of clobberize() can
112 // just ignore def() by using a NoOpClobberize functor.
113
114 // We allow the runtime to perform a stack scan at any time. We don't model which nodes get implemented
115 // by calls into the runtime. For debugging we might replace the implementation of any node with a call
116 // to the runtime, and that call may walk stack. Therefore, each node must read() anything that a stack
117 // scan would read. That's what this does.
118 for (InlineCallFrame* inlineCallFrame = node->origin.semantic.inlineCallFrame(); inlineCallFrame; inlineCallFrame = inlineCallFrame->directCaller.inlineCallFrame()) {
119 if (inlineCallFrame->isClosureCall)
120 read(AbstractHeap(Stack, VirtualRegister(inlineCallFrame->stackOffset + CallFrameSlot::callee)));
121 if (inlineCallFrame->isVarargs())
122 read(AbstractHeap(Stack, VirtualRegister(inlineCallFrame->stackOffset + CallFrameSlot::argumentCountIncludingThis)));
123 }
124
125 // We don't want to specifically account which nodes can read from the scope
126 // when the debugger is enabled. It's helpful to just claim all nodes do.
127 // Specifically, if a node allocates, this may call into the debugger's machinery.
128 // The debugger's machinery is free to take a stack trace and try to read from
129 // a scope which is expected to be flushed to the stack.
130 if (graph.hasDebuggerEnabled()) {
131 ASSERT(!node->origin.semantic.inlineCallFrame());
132 read(AbstractHeap(Stack, graph.m_codeBlock->scopeRegister()));
133 }
