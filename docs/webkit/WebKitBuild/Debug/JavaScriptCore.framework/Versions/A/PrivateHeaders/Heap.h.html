193 // of equal or greater strength exists. Full collections are stronger than std::nullopt collections

322 // If true, the GC believes that the mutator is currently messing with the heap. We call this
323 // "having heap access". The GC may block if the mutator is in this state. If false, the GC may
324 // currently be doing things to the heap that make the heap unsafe to access for the mutator.
325 bool hasAccess() const;
326
327 // If the mutator does not currently have heap access, this function will acquire it. If the GC
328 // is currently using the lack of heap access to do dangerous things to the heap then this
329 // function will block, waiting for the GC to finish. It's not valid to call this if the mutator
330 // already has heap access. The mutator is required to precisely track whether or not it has
331 // heap access.
332 //
333 // It's totally fine to acquireAccess() upon VM instantiation and keep it that way. This is how
334 // WebCore uses us. For most other clients, JSLock does acquireAccess()/releaseAccess() for you.
335 void acquireAccess();
336
337 // Releases heap access. If the GC is blocking waiting to do bad things to the heap, it will be
338 // allowed to run now.
339 //
340 // Ordinarily, you should use the ReleaseHeapAccessScope to release and then reacquire heap
341 // access. You should do this anytime you're about do perform a blocking operation, like waiting
342 // on the ParkingLot.
343 void releaseAccess();
344
345 // This is like a super optimized way of saying:
346 //
347 // releaseAccess()
348 // acquireAccess()
349 //
350 // The fast path is an inlined relaxed load and branch. The slow path will block the mutator if
351 // the GC wants to do bad things to the heap.
352 //
353 // All allocations logically call this. As an optimization to improve GC progress, you can call
354 // this anywhere that you can afford a load-branch and where an object allocation would have been
355 // safe.
356 //
357 // The GC will also push a stopIfNecessary() event onto the runloop of the thread that
358 // instantiated the VM whenever it wants the mutator to stop. This means that if you never block
359 // but instead use the runloop to wait for events, then you could safely run in a mode where the
360 // mutator has permanent heap access (like the DOM does). If you have good event handling
361 // discipline (i.e. you don't block the runloop) then you can be sure that stopIfNecessary() will
362 // already be called for you at the right times.
363 void stopIfNecessary();
