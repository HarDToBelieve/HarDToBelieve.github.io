72// This represents B3's view of a piece of code. Note that this object must exist in a 1:1
73// relationship with Air::Code. B3::Procedure and Air::Code are just different facades of the B3
74// compiler's knowledge about a piece of code. Some kinds of state aren't perfect fits for either
75// Procedure or Code, and are placed in one or the other based on convenience. Procedure always
76// allocates a Code, and a Code cannot be allocated without an owning Procedure and they always
77// have references to each other.
78

105 // Changes the order of basic blocks to be as in the supplied vector. The vector does not
106 // need to mention every block in the procedure. Blocks not mentioned will be placed after
107 // these blocks in the same order as they were in originally.
108 template<typename BlockIterable>
109 void setBlockOrder(const BlockIterable& iterable)
110 {
111 Vector<BasicBlock*> blocks;
112 for (BasicBlock* block : iterable)
113 blocks.append(block);
114 setBlockOrderImpl(blocks);
115 }

215 // Some operations are specified in B3 IR to behave one way but on this given CPU they behave a
216 // different way. When true, those B3 IR ops switch to behaving the CPU way, and the optimizer may
217 // start taking advantage of it.
218 //
219 // One way to think of it is like this. Imagine that you find that the cleanest way of lowering
220 // something in lowerMacros is to unconditionally replace one opcode with another. This is a shortcut
221 // where you instead keep the same opcode, but rely on the opcode's meaning changes once lowerMacros
222 // sets hasQuirks.
223 bool hasQuirks() const { return m_hasQuirks; }
224 void setHasQuirks(bool value) { m_hasQuirks = value; }

226 OpaqueByproducts& byproducts() { return *m_byproducts; }

228 // Below are methods that make sense to call after you have generated code for the procedure.
229
230 // You have to call this method after calling generate(). The code generated by B3::generate()
231 // will require you to keep this object alive for as long as that code is runnable. Usually, this
232 // just keeps alive things like the double constant pool and switch lookup tables. If this sounds
233 // confusing, you should probably be using the JSC::Compilation API to compile code. If you use
234 // that API, then you don't have to worry about this.
235 std::unique_ptr<OpaqueByproducts> releaseByproducts() { return WTFMove(m_byproducts); }
