44// A ParallelHelperPool is a shared pool of threads that can be asked to help with some finite-time
45// parallel activity. It's designed to work well when there are multiple concurrent tasks that may
46// all want parallel help. In that case, we don't want each task to start its own thread pool. It's
47// also designed to work well for tasks that do their own load balancing and do not wish to
48// participate in microtask-style load balancing.
49//
50// A pool can have many clients, and each client may have zero or one tasks. The pool will have up
51// to some number of threads, configurable with ParallelHelperPool::addThreads(); usually you bound
52// this by the number of CPUs. Whenever a thread is idle and it notices that some client has a
53// task, it will run the task. A task may be run on anywhere between zero and N threads, where N is
54// the number of threads in the pool. Tasks run to completion. It's expected that a task will have
55// its own custom ideas about how to participate in some parallel activity's load balancing, and it
56// will return when the parallel activity is done. For example, a parallel marking task will return
57// when the mark phase is done.
58//
59// Threads may have a choice between many tasks, since there may be many clients and each client
60// may have a task. For the marking example, that may happen if there are multiple VM instances and
61// each instance decides to start parallel marking at the same time. In that case, threads choose
62// a task at random. So long as any client has a task, all threads in the pool will continue
63// running the available tasks. Threads go idle when no client has tasks to run.
64
65class ParallelHelperClient;

101// A client is a placeholder for a parallel algorithm. A parallel algorithm will have a task that
102// can be run concurrently. Whenever a client has a task set (you have called setTask() or
103// setFunction()), threads in the pool may run that task. If a task returns on any thread, the
104// client will assume that the task is done and will clear the task. If the task is cleared (the
105// task runs to completion on any thread or you call finish()), any threads in the pool already
106// running the last set task(s) will continue to run them. You can wait for all of them to finish
107// by calling finish(). That method will clear the task and wait for any threads running the last
108// set task to finish. There are two known-good patterns for using a client:
109//
110// 1) Tasks intrinsically know when the algorithm reaches termination, and simply returns when
111// this happens. The main thread runs the task by doing:
112//
113// client->setFunction(
114// [=] () {
115// do things;
116// });
117// client->doSomeHelping();
118// client->finish();
119//
120// Calling doSomeHelping() ensures that the algorithm runs on at least one thread (this one).
121// Tasks will know when to complete, and will return when they are done. This will clear the
122// task to ensure that no new threads will run the task. Then, finish() clears the current task
123// and waits for any parallel tasks to finish after the main thread has finished. It's possible
124// for threads to still be running the last set task (i.e. the one set by setFunction()) even
125// after the task has been cleared. Waiting for idle ensures that no old tasks are running
126// anymore.
127//
128// You can do this more easily by using the runFunctionInParallel() helper:
129//
130// clients->runFunctionInParallel(
131// [=] () {
132// do things;
133// });
134//
135// 2) Tasks keep doing things until they are told to quit using some custom notification mechanism.
136// The main thread runs the task by doing:
137//
138// bool keepGoing = true;
139// client->setFunction(
140// [=] () {
141// while (keepGoing) {
142// do things;
143// }
144// });
145//
146// When work runs out, the main thread will inform tasks that there is no more work, and then
147// wait until no more tasks are running:
148//
149// keepGoing = false;
150// client->finish();
151//
152// This works best when the main thread doesn't actually want to run the task that it set in the
153// client. This happens for example in parallel marking. The main thread uses a somewhat
154// different marking algorithm than the helpers. The main thread may provide work that the
155// helpers steal. The main thread knows when termination is reached, and simply tells the
156// helpers to stop upon termination.
157//
158// The known-good styles of using ParallelHelperClient all involve a parallel algorithm that has
159// its own work distribution and load balancing.
160//
161// Note that it is not valid to use the same ParallelHelperClient instance from multiple threads.
162// Each thread should have its own ParallelHelperClient in that case. Failure to follow this advice
163// will lead to RELEASE_ASSERT's or worse.
164class ParallelHelperClient {
165 WTF_MAKE_NONCOPYABLE(ParallelHelperClient);
166 WTF_MAKE_FAST_ALLOCATED;
167public:
168 WTF_EXPORT_PRIVATE ParallelHelperClient(RefPtr<ParallelHelperPool>&&);
